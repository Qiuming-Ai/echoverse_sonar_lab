# ESL3D Binary File Format Specification (v1)

## Scope

This document defines the **exact on-disk binary format** of `.esl3d` files as produced by the current EchoVerse Sonar Lab implementation.

An `.esl3d` file is an append-only sequence of packets.  
Each packet has the same binary layout as one TCP point-cloud packet generated by `PointCloudTcpStreamer::buildPacket(...)`.

## Endianness

- All numeric fields in the fixed header are **little-endian**.
- `range` and `intensity` payload arrays are stored as little-endian IEEE-754 `float32`.

## File Structure

The file is a concatenation of zero or more packets:

1. Fixed header (`56` bytes)
2. Metadata JSON (`metadata_bytes`)
3. Range image (`range_bytes`)
4. Intensity image (`intensity_bytes`)

No global file header is used.

## Packet Header (56 bytes)

| Offset | Size | Type | Field | Value / Meaning |
|---|---:|---|---|---|
| 0 | 4 | `u32` | `magic` | Constant `0x5033534E` (`"NS3P"`) |
| 4 | 2 | `u16` | `version` | Current version is `1` |
| 6 | 2 | `u16` | `header_bytes` | Must be `56` |
| 8 | 8 | `u64` | `seq` | Monotonic sequence number |
| 16 | 8 | `u64` | `timestamp_us` | Timestamp in microseconds |
| 24 | 4 | `u32` | `width` | Polar image width |
| 28 | 4 | `u32` | `height` | Polar image height |
| 32 | 4 | `u32` | `point_count` | Usually `width * height` |
| 36 | 4 | `u32` | `metadata_bytes` | Metadata payload size |
| 40 | 4 | `u32` | `range_bytes` | Range payload size |
| 44 | 4 | `u32` | `intensity_bytes` | Intensity payload size |
| 48 | 4 | `u32` | `payload_bytes` | `metadata_bytes + range_bytes + intensity_bytes` |
| 52 | 4 | `u32` | `reserved` | Currently `0` |

## Payload Layout

Immediately after the 56-byte header:

1. `metadata_bytes` bytes of UTF-8 JSON (compact JSON text)
2. `range_bytes` bytes of float data
3. `intensity_bytes` bytes of float data

Payload slicing rules:

- `metadata = payload[0 : metadata_bytes]`
- `range = payload[metadata_bytes : metadata_bytes + range_bytes]`
- `intensity = payload[metadata_bytes + range_bytes : payload_bytes]`

## Array Encoding

- `range` contains `float32` values in row-major order.
- `intensity` contains `float32` values in row-major order.
- Expected element count for each image is `width * height`.
- Therefore:
  - `range_bytes` should be `4 * width * height`
  - `intensity_bytes` should be `4 * width * height`

When reconstructing 2D arrays:

- Shape is `(height, width)`.
- Linear index is `idx = row * width + col`.

## Metadata JSON Contract

Metadata is JSON and may evolve, but current writer includes:

- `byte_order`: `"little_endian"`
- `layout`: `"row_major"`
- `data_order`: `"range_then_intensity"`
- `range_invalid_value`
- `intensity_invalid_value`
- `frame` object:
  - `seq`
  - `timestamp_us`
  - `width`
  - `height`
  - `point_count`
  - `rounding_policy` (`"floor"`)
- `pose` object:
  - `x`, `y`, `z`
  - `yaw_deg`, `pitch_deg`
  - `quat_w`, `quat_x`, `quat_y`, `quat_z`
- `sonar_config` object (runtime config snapshot)
- `environment` object (runtime environment snapshot)

## Validation Rules (Recommended)

A robust parser should validate at least:

1. `magic == 0x5033534E`
2. `version == 1`
3. `header_bytes == 56`
4. `payload_bytes == metadata_bytes + range_bytes + intensity_bytes`
5. `range_bytes % 4 == 0` and `intensity_bytes % 4 == 0`
6. `range_bytes / 4 == width * height` (strict mode)
7. `intensity_bytes / 4 == width * height` (strict mode)
8. EOF is reached exactly at packet boundaries (or clean end-of-file)

## Reference Binary Struct

```c
// little-endian, packed, 56 bytes total
struct Esl3dHeaderV1 {
    uint32_t magic;           // 0x5033534E
    uint16_t version;         // 1
    uint16_t header_bytes;    // 56
    uint64_t seq;
    uint64_t timestamp_us;
    uint32_t width;
    uint32_t height;
    uint32_t point_count;
    uint32_t metadata_bytes;
    uint32_t range_bytes;
    uint32_t intensity_bytes;
    uint32_t payload_bytes;
    uint32_t reserved;        // 0
};
```

## Streaming Parse Procedure

1. Read 56-byte header.
2. Decode fields as little-endian integers.
3. Validate header fields.
4. Read `payload_bytes`.
5. Slice into metadata/range/intensity by length fields.
6. Decode metadata JSON (UTF-8).
7. Reinterpret range/intensity bytes as little-endian `float32`.
8. Reshape arrays to `(height, width)` in row-major order.
9. Repeat until clean EOF.

## Compatibility Notes

- This specification is tied to current writer behavior (`version = 1`).
- Future versions should bump `version` and keep backward parsing logic.
- Parsers should use `header_bytes` and length fields, not hard-coded payload assumptions beyond v1 validation.
